How Trajectory-Only Replay Works

  • Zero Video Buffering: Only stores trajectory points (TrajectoryPoint[]) - typically 20-30 points × ~40 bytes = <1KB
  • Optional Static Frame: Can capture one single frame at ball release for context (640×360 = ~900KB)
  • Animated Visualization: Replay shows ball movement along trajectory path using canvas animation
  • Multiple View Options: Side view (Hawk-Eye), top-down (wagon wheel), 3D isometric, or camera perspective
  • Professional Look: Mimics TV broadcast graphics that cricket fans are familiar with

✅ Advantages

  • Zero Memory Issues: ~1KB for trajectory + ~900KB for optional static frame
  • Professional Look: Hawk-Eye/DRS style visualization
  • Fast Export: JSON is tiny, can generate video from trajectory
  • Works Everywhere: No memory constraints, even on old phones
  • Analytical Focus: Shows path, speed zones, bounce point clearly
  • Easy Sharing: Can render to image/video for social media

⚠️ Trade-offs

  • No Bowler Action: Can't see delivery technique or follow-through
  • No Batsman Context: Can't see where batsman was, field placement
  • No Environmental Context: Miss lighting, pitch condition visuals
  • Expectation Gap: Users might expect full video replay
  • Less "Wow" Factor: Not as engaging as video replay initially

💡 Hybrid Solution: Best of Both Worlds

  • Default: Trajectory-Only for instant, memory-safe replay on all devices
  • Optional: Video Buffer as opt-in feature for high-end devices (with memory check)
  • Export Options: Always provide trajectory animation export, optionally include video if buffered
  • Progressive Enhancement: Start with trajectory, add video if device capable
